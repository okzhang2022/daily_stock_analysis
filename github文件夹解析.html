.github 文件夹解析

这份文件专门解释 .github/ 目录：它主要承载 CI/CD 工作流、Issue/PR 模板、仓库治理指令 与 自动化脚本。

入口：.github/workflows/*.yml PR 审查：.github/workflows/pr-review.yml + .github/scripts/ai_review.py 发布：tag → release / docker / desktop 治理：instructions + copilot-instructions

总览：.github 下有什么

路径 作用 关键点
.github/workflows/ GitHub Actions 工作流：CI、每日任务、发布、Docker 构建、PR 自动审查、清理 stale 等。 是最核心的自动化入口；每个 yml 文件是一条独立流水线。
.github/scripts/ 工作流用的脚本（主要是 PR AI 审查）。 ai_review.py 会被 PR Review 工作流调用，收集 diff + PR 描述后调用 LLM。
.github/instructions/ Copilot/Coding Agent 的路径级指令（backend/client/governance）。 用 applyTo 指定哪些路径受哪些规则约束，规则与仓库 AGENTS.md 对齐。
.github/ISSUE_TEMPLATE/ Issue 模板：Bug/Feature 等，统一收集复现信息与环境。 引导用户贴 commit hash、运行方式、关键配置与日志，减少来回沟通。
.github/PULL_REQUEST_TEMPLATE.md PR 模板：类型、背景、范围、验证、风险、回滚等字段。 强调“给出实际跑过的验证命令”，并对 prompt 变更（EXTRACT_PROMPT）做了特殊要求。
.github/CODEOWNERS 代码所有者：指定默认审阅者/负责人。 当前为全仓库默认 owner。
.github/copilot-instructions.md Copilot 仓库指令汇总（镜像 AGENTS.md）。 明确“以 AGENTS.md 为真源”，并列出目录边界、验证矩阵与治理规则。
.github/release.yml GitHub Release notes 分类规则（changelog categories）。 用 label 将 PR 自动归类到 New Features / Bug Fixes / Docs 等类别。
.github/FUNDING.yml 赞助入口配置。 指向 README 的 sponsor 段落。

.github/instructions：三类指令分别约束什么

backend.instructions.md
适用：main.py/server.py/src/**/data_provider/**/api/**/bot/**/tests/**
重点：保持流水线边界、配置变更要评估多端影响、data_provider 改动必须保留 fallback/降级语义、优先跑 ./scripts/ci_gate.sh。
client.instructions.md
适用：apps/dsa-web/**、apps/dsa-desktop/** 及相关脚本/桌面文档
重点：保持 Vite+React 与 Electron 既有结构；API 字段/认证/渲染改动要评估 Web+Desktop；前端要跑 lint+build。
governance.instructions.md
适用：README/docs/.github/scripts/docker 等治理与发布路径
重点：AGENTS.md 真源、避免权限扩大/密钥暴露、保持 auto-tag 只在 #patch/#minor/#major 时触发。

.github/scripts：自动审查脚本做什么

ai_review.py：被 PR Review 工作流调用。它会：
筛选范围
只审查特定路径（py/md/docs/workflows/web 等），避免把不相关 diff 喂给模型。
diff 收集
对 base 分支做 diff，超长会截断（MAX_DIFF_LENGTH）。
提示词
按 AGENTS.md 的审查维度输出：必要性/关联性/类型/描述完整性/风险/必改项/建议项。
模型调用
优先 Gemini（依赖 GEMINI_API_KEY 等环境变量）。
注意：该脚本用于“评论式审查”，不是阻断 CI 的 gate；真正阻断类检查主要在 CI 工作流（ci.yml）里完成。

.github/workflows：每个工作流文件做什么

这里按文件逐个解释“触发条件 / 主要工作 / 产物”。你可以把它理解成一组互不相同的流水线：有的面向 PR，有的面向每日运行，有的面向发布与打包。
ci.yml — PR CI Gate pull_request
触发
PR 提交到 main（pull_request）。
目标
保证后端、Docker、（可选）前端改动的基础质量与可构建性。
关键 job
ai-governance（校验 AI 协作资产），backend-gate（syntax/flake8/deterministic/offline-tests），docker-build（构建镜像+import smoke），web-gate（仅当前端路径变更时运行）。
pr-review.yml — PR 自动审查（静态 + AI + 自动评论/打标签） pull_request_target
触发
pull_request_target（opened/synchronize/reopened），并限定一组路径；也支持 workflow_dispatch。
做什么
1) 安全检查（是否改了 workflow/scripts 等敏感文件）→ 2) 静态检查（py_compile + flake8 严重错误）→ 3) AI 审查（调用 .github/scripts/ai_review.py）→ 4) 自动打标签 → 5) 汇总评论到 PR。
权限
需要对 PR/Issue 写权限以评论与打标签。
为什么用 pull_request_target？ 为了让工作流在 fork PR 也能拿到写评论/打标签权限。风险点是“执行 PR 代码时可能有安全隐患”，该工作流通过先检出主分支脚本、再检出 PR 代码并限制路径/步骤来降低风险，但对修改 workflow 的 PR 仍建议人工重点复核。
daily_analysis.yml — 每日股票分析（定时 + 手动） schedule
触发
工作日定时（UTC 10:00 = 北京 18:00）+ workflow_dispatch。
模式
full（股票+大盘）、market-only（仅大盘复盘：python main.py --market-review）、 stocks-only（仅个股：python main.py --no-market-review）。
产物
上传 reports/ 与 logs/ 为 artifact，并把最新报告 md 转成 reports/index.html 发布到 GitHub Pages。
依赖
大量从 GitHub Secrets/Vars 注入 LLM 与数据源配置（Gemini/OpenAI/LiteLLM/Longbridge/Tushare 等）。
network-smoke.yml — 网络相关冒烟（非阻断） schedule
触发
工作日定时（UTC 2:00）+ 手动触发。
做什么
运行 pytest -m network 与 ./scripts/test.sh quick，都设置为 continue-on-error，不阻断。
产物
上传 network smoke 的日志 artifact，方便排查网络与外部依赖波动。
auto-tag.yml — 自动打版本 Tag（opt-in） push main
触发
push 到 main（忽略 docs/md/.github 等路径）。
策略
默认不打 tag；只有 commit message 含 #patch/#minor/#major 才会触发。
实现
使用 anothrNick/github-tag-action 自动 bump 并 push tag。
create-release.yml — 从 Tag 创建 GitHub Release push tag
触发
push annotated tag：v*.*.*。
关键点
读取 tag 的 annotation 内容作为 release body（如果为空则用默认文案）。
实现
使用 softprops/action-gh-release 创建 Release。
desktop-release.yml — 桌面端打包与发布（Windows/macOS） push tag / manual
触发
push tag（vX.Y.Z）或手动输入 release_tag。
构建矩阵
Windows installer + macOS x64/arm64 dmg（分别在对应 runner 上打包）。
产物
上传安装包/免安装 zip/dmg，再汇总发布到 GitHub Release。
说明
发布正文优先从 docs/CHANGELOG.md 中提取对应版本段落。
docker-publish.yml — Docker Release Publish（随 tag 发布） push tag / manual
触发
push tag（vX.Y.Z）或手动输入 release_tag。
前置 gate
发布前会跑 ./scripts/ci_gate.sh，并要求 tag 必须是 annotated（annotation 不为空）。
发布目标
推送多架构镜像到 GHCR；若配置 DockerHub token/username，也同步推到 Docker Hub。
ghcr-dockerhub.yml — Docker Manual Publish（完全手动） manual only
触发
仅 workflow_dispatch，输入 image_tag。
用途
不依赖 tag 事件，适合临时补发镜像或回滚镜像 tag。
发布
GHCR + Docker Hub 都会推送，带 pre-publish smoke。
stale.yml — 自动标记/关闭长期无活动的 Issue/PR schedule
触发
每日 UTC 0:00 + 手动触发。
规则
Issue 7 天无活动标 stale，再 7 天关闭；PR 21 天标 stale，再 14 天关闭；排除 pinned/security 等标签与草稿 PR。
实现
actions/stale。
读法建议：先看 ci.yml（合入质量门），再看 daily_analysis.yml（真实运行路径），最后看发布链路（auto-tag → create-release / docker-publish / desktop-release）。

你最可能会用到的定位入口

想知道“PR 合入前会跑什么”
先看 .github/workflows/ci.yml（阻断 CI）与 .github/workflows/pr-review.yml（自动审查/评论）。
想知道“每天自动跑分析到底怎么跑的”
看 .github/workflows/daily_analysis.yml，里面明确了 full/market-only/stocks-only 三种运行命令以及 Pages 发布。
想知道“发版会发生什么”
tag 流：auto-tag.yml（可选）→ create-release.yml（Release）→ docker-publish.yml / desktop-release.yml（产物发布）。
改 workflow 的注意事项
workflow 本身属于“敏感变更”，PR Review 会提示重点人工复核；同时要避免扩大权限、避免泄露 secrets，并确保仍符合仓库的 opt-in tag 策略。
