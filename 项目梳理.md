# daily_stock_analysis 项目梳理

## 1. 这个项目是做什么的

这是一个面向 **A 股 / 港股 / 美股** 的“自选股智能分析系统”。它把「行情数据 + 技术指标 + 多维度情报（新闻/风险/事件等）+ LLM 归纳」串成一条可自动化的流水线，最终产出可读、可运营的 **决策仪表盘式报告**，并支持多渠道推送、API/Web 交互、Bot 触达与（可选）桌面端承载。

常见使用场景：
- 每日/每次运行：批量分析一组股票并生成汇总报告（`reports/report_YYYYMMDD.md`）
- 盘后复盘：按市场生成“大盘复盘”（`reports/market_review_YYYYMMDD.md`）
- Web/API：通过 FastAPI 提供分析、历史、组合等接口，WebUI 展示与触发
- Bot：通过命令触发分析/复盘/查询

关键入口：
- CLI/调度入口：[main.py](file:///Users/didi/Documents/didi/github/daily_stock_analysis/main.py)
- API 服务入口：[server.py](file:///Users/didi/Documents/didi/github/daily_stock_analysis/server.py)

## 2. 目标是什么

面向“可用的交易辅助”而不是“单点分析 demo”，目标可概括为：
- **自动化**：减少手工查价/翻新闻/做笔记的成本，支持定时任务与批量处理
- **多市场**：同一套流程兼容 cn/hk/us，尽量减少硬编码与市场耦合
- **稳定可运行**：外部数据源不稳定时可降级；单股失败不拖垮全局；推送失败不影响主流程
- **可交互可集成**：CLI、API/Web、Bot 等多入口复用同一核心能力
- **输出可行动**：报告结构偏“决策仪表盘”，强调结论、风险、观察点与条件化计划

## 4. 核心流程（从触发到报告/推送）

### 4.1 个股批量分析（报告：report_YYYYMMDD.md）

典型路径：
- `main.py` 解析股票列表/运行模式 → 创建 `StockAnalysisPipeline`
- `StockAnalysisPipeline` 批量并发执行：
  - 数据抓取（多数据源 + fallback）
  - 情报搜索（新闻/风险/事件）
  - LLM 分析（结构化“仪表盘”）
  - 保存报告文件 +（可选）推送通知

对应核心实现：
- 编排入口：[main.py](file:///Users/didi/Documents/didi/github/daily_stock_analysis/main.py)
- 流水线实现：[src/core/pipeline.py](file:///Users/didi/Documents/didi/github/daily_stock_analysis/src/core/pipeline.py)
- 报告生成与落盘：通知层的 `save_report_to_file`：[src/notification.py](file:///Users/didi/Documents/didi/github/daily_stock_analysis/src/notification.py#L1722-L1753)

### 4.2 大盘复盘（报告：market_review_YYYYMMDD.md）

复盘与个股分析是独立链路，按配置决定市场区域（`MARKET_REVIEW_REGION`）。
- 复盘编排：[src/core/market_review.py](file:///Users/didi/Documents/didi/github/daily_stock_analysis/src/core/market_review.py)
- 复盘 Prompt 与生成：[src/market_analyzer.py](file:///Users/didi/Documents/didi/github/daily_stock_analysis/src/market_analyzer.py)

### 4.3 多入口复用同一核心

项目的一个关键点是：CLI / API / Bot 看似入口不同，但尽量复用同一套核心能力（流水线 + 服务层），降低“多入口导致多套实现发散”的维护成本。

## 5. 设计理念与取舍

从实现和目录边界可以归纳为以下几条“默认价值观”：
- **稳定性优先**：外部数据源与搜索是高不确定性依赖，系统倾向“可降级继续产出”，而不是 fail-fast
- **异常隔离**：单股失败不影响其他股票；通知/落盘失败不影响分析结果产出
- **配置驱动**：市场区域、是否启用复盘、交易日过滤、推送渠道等通过配置控制，支持不同部署形态
- **输出结构化**：以“决策仪表盘”的结构组织结论，便于前端展示、推送裁剪、后续扩展为 API Schema
- **分层复用**：入口层薄、核心编排集中、外部能力通过适配器封装，尽量避免跨层耦合

## 7. 一句话总结

这是一个“把多市场股票分析做成可持续跑、可多端触达、可降级容错”的系统：入口多样，但核心围绕同一条流水线产出统一格式的决策报告，并通过存储与通知形成闭环。
